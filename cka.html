Kubernetes Architecture Overview

The fundamental separation of duties in a K8s cluster.

🧠

Control Plane
Master Node

The Control Plane manages the worker nodes and the Pods in the cluster. In production environments, the control plane usually runs across multiple computers for high availability.

kube-apiserver

The front end of the control plane. ALL communication (from kubectl, workers, etc.) goes through the API server.

etcd

A highly-available, consistent key-value store holding the cluster's entire state. If etcd is lost, the cluster is lost. Backing up etcd is a major CKA requirement.

kube-scheduler & controller-manager

**Scheduler:** Watches for new Pods with no assigned node, and assigns them one based on resources/taints. **Controller-Manager:** Runs controllers like the Node controller (notices when nodes go down) and ReplicaSet controller.

🖥️

Worker Nodes
Compute Instances

Node components run on every node, maintaining running pods and providing the Kubernetes runtime environment.

kubelet

The primary "node agent" that runs on each node. It takes PodSpecs (provided by the API server) and ensures that the containers described in those PodSpecs are running and healthy.

kube-proxy

A network proxy that runs on each node. It maintains network rules (usually via iptables) that allow network communication to your Pods from inside or outside of your cluster.

Container Runtime

The software that actually runs the containers (e.g., containerd, CRI-O). Docker is no longer supported directly as a runtime starting in K8s v1.24.
