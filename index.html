Shared Responsibility Model

A comprehensive breakdown of Security "OF" the Cloud vs. Security "IN" the Cloud.

👤

Customer
Security "IN" the Cloud

The customer assumes responsibility and management of the guest operating system (including updates and security patches), other associated application software, as well as the configuration of the AWS-provided security group firewall.

Customer Data

You retain absolute control over your data. You are responsible for managing data classification, client-side data encryption, server-side encryption (file system and/or data), and network traffic protection (encryption, integrity, identity).

IAM & Access Control

Configuring robust Identity and Access Management (IAM), enforcing Multi-Factor Authentication (MFA), setting password policies, and defining strict resource-level permissions.

OS, Network & Firewall

Patching and maintaining the Guest Operating System (e.g., EC2 instances), configuring Security Groups (virtual firewalls), and setting up Network Access Control Lists (NACLs).

☁️

AWS
Security "OF" the Cloud

AWS is responsible for protecting the infrastructure that runs all of the services offered in the AWS Cloud. This infrastructure is composed of the hardware, software, networking, and facilities.

Global Infrastructure

Physical security and environmental controls of AWS Regions, Availability Zones (AZs), and Edge Locations. This includes power, cooling, and perimeter security.

Foundational Compute & Storage

Maintaining and patching the physical hypervisors, storage arrays, and core networking hardware that power services like Amazon EC2, EBS, and VPC.

Managed Services Software

For abstracted services like Amazon RDS, S3, or DynamoDB, AWS assumes additional responsibility for operating system patching, database software updates, and underlying infrastructure maintenance.

💡 Exam Tip: As you move to managed services (like Lambda or S3), the customer responsibility shifts upward, but you are ALWAYS responsible for your Data and IAM.
